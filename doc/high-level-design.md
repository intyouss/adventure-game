## 三、核心模块设计

### 3.1 账号模块

**功能**：
- 注册：支持手机号 + 密码 / 邮箱 + 密码
- 登录：验证凭证，返回 JWT Token
- Token 刷新：Access Token 过期后用 Refresh Token 换新

**Token 机制**（按常规做法 — JWT 双 Token）：

| 类型 | 有效期 | 用途 |
|------|--------|------|
| Access Token | 2 小时 | 业务请求鉴权 |
| Refresh Token | 7 天 | 换取新 Access Token |

**鉴权流程**：
1. 客户端登录 → 服务端返回 `access_token` + `refresh_token`
2. 后续请求在 Header 携带 `Authorization: Bearer <access_token>`
3. Access Token 过期 → 客户端用 `refresh_token` 调 `/auth/refresh` 换取新 Token
4. Refresh Token 也过期 → 需重新登录

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/register` | 注册 |
| POST | `/api/auth/login` | 登录 |
| POST | `/api/auth/refresh` | 刷新 Token |

### 3.2 角色模块

**功能**：
- 管理 1 个职业（第一版），职业决定固定技能和基础属性
- 等级提升：消耗经验值，指数曲线增长
- 基础属性：攻击力、防御力、生命值，随等级线性增长
- 经验获取：装备分解

**经验曲线**（指数增长）：
```
ExpToNextLevel(L) = BaseExp × GrowthRate ^ (L - 1)
```
- `BaseExp`：升到 2 级所需经验（建议 100）
- `GrowthRate`：增长系数（建议 1.15）

**装备分解经验/金币**：

| 品质 | 经验 | 金币 |
|------|------|------|
| 普通（白） | 10 | 50 |
| 优秀（绿） | 25 | 120 |
| 稀有（蓝） | 60 | 300 |
| 精良（紫） | 140 | 700 |
| 史诗（橙） | 350 | 1,800 |
| 传说（红） | 800 | 4,500 |
| 神话（彩） | 2,000 | 12,000 |

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/character` | 获取角色信息 |
| POST | `/api/character/add_exp` | 增加经验（关卡结算/分解触发） |

### 3.3 技能模块

**功能**：
- 技能仓库：存储玩家拥有的技能及其品质
- Gacha 抽取：消耗技能券，支持单抽和多抽（10 连）

- 技能槽位：1 个固定职业技能 + 4 个可选技能，不可重复装备同一技能
- 所有技能全职业通用

**商店等级与品质解锁**（初始仅白、蓝可抽取）：

| 商店等级 | 解锁品质 | 可抽取品质池 |
|----------|----------|--------------|
| Lv1-7    | —        | 白、蓝 |
| Lv8-15   | 紫       | 白、蓝、紫 |
| Lv16-23  | 橙       | 白、蓝、紫、橙 |
| Lv24-28  | 红       | 白、蓝、紫、橙、红 |

- 商店最高 **28 级**，不会禁用低品质
- 商店等级根据**累计抽取次数**升级

**抽取概率（按固定权重比例分配）**：
- 各品质在活跃池内按固定权重随机，高品质权重低于低品质
- 解锁新品质后，总权重重新分配
- 无保底机制

**多抽支持**：单抽 / 10 连 / 50 连 / 100 连（参数 `count` ∈ {1, 10, 50, 100}）
- 多抽等效于多次独立抽取

**技能升星系统**：
- 技能最高 **30 级**，消耗同名技能卡升级
- 消耗数量：

| 升级 | 消耗张数 |
|------|----------|
| L1→L2 | 1 |
| L2→L3 | 1 |
| L3→L4 | 3 |
| L4→L5 | 5 |
| L5→L6 | 7 |
| L6→L7 | 11 |
| L7→L8 | 16 |
| ... | 指数递增 |

- 计算公式（N≥4 为目标等级）：`消耗张数 = ceil(3 × 1.5^(N-3))`
- 技能品质与升星消耗无关（所有品质升星消耗相同）
- 技能等级影响技能系数，等级越高系数越大

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/skill/list` | 获取技能仓库 |
| POST | `/api/skill/gacha` | 抽取技能（count ∈ {1, 10, 50, 100}） |
| POST | `/api/skill/equip` | 装备技能到槽位 |
| GET | `/api/skill/shop_info` | 获取商店等级和概率 |
| POST | `/api/skill/upgrade` | 消耗同名卡升级技能 |

### 3.4 装备模块

**功能**：
- 装备仓库：存储玩家拥有的全部装备
- 装备栏：10 个槽位（武器/头盔/衣服/鞋子/戒指1/戒指2/项链/护腕/腰带/手套）
- 装备/卸下：装备放入槽位立即生效属性，卸下后属性还原
- 分解：消耗装备获得经验和金币，品质越高收益越大

**装备属性模型**：
```
装备 = {
  唯一ID, 部位, 品质, 攻击加成, 防御加成, 生命加成
}
```
- 同品质装备属性在数值范围内随机生成
- 无强化系统，获得时属性即确定
- 各部位属性类型：

| 部位 | 属性类型 |
|------|----------|
| 武器 | 攻击、攻速、暴击率 |
| 头盔 | 生命、防御 |
| 衣服 | 生命、防御 |
| 鞋子 | 生命、移动速度 |
| 戒指 1/2 | 攻击、暴击率 |
| 项链 | 生命、暴击伤害 |
| 护腕 | 攻击、防御 |
| 腰带 | 生命、防御 |
| 手套 | 攻击、攻速、暴击率 |

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/equipment/inventory` | 获取装备仓库 |
| POST | `/api/equipment/equip` | 装备到槽位 |
| POST | `/api/equipment/unequip` | 卸下装备 |
| POST | `/api/equipment/decompose` | 分解装备 |

### 3.5 开箱模块

**功能**：
- 箱子积攒：关卡掉落箱子存入玩家背包
- 开箱：消耗箱子，开箱时随机品质，受区域等级约束
- 区域等级：独立升级系统，消耗金币

**区域等级与品质解锁规则**：

| 区域等级 | 解锁品质 | 活跃品质池 |
|----------|----------|------------|
| Lv1-4    | —        | 白、绿 |
| Lv5-9    | 蓝       | 白、绿、蓝 |
| Lv10-14  | 紫       | 白、绿、蓝、紫 |
| Lv15-19  | 橙       | 绿、蓝、紫、橙（白被禁用） |
| Lv20-24  | 红       | 蓝、紫、橙、红（绿被禁用） |
| Lv25-27  | 彩       | 紫、橙、红、彩（蓝被禁用） |
| Lv28     | —        | 橙、红、彩 + 最高品质固定 5%（传说池缩为 3 品质） |

- 开箱区域最高 **28 级**
- 规则：每 5 级解锁一个新品质；活跃品质池保持 4 个；溢出时禁用最低品质
- **Lv28 满级后**：最高品质（神话/彩色）固定抽取概率 **5%**，橙色/红色/彩色三者内部分配

**开箱随机权重**（以活跃 4 品质为例）：

| 品质 | 权重 |
|------|------|
| 最低品质 | 40% |
| 第二品质 | 30% |
| 第三品质 | 20% |
| 最高品质 | 10% |

**区域升级消耗**（指数增长）：
```
升级费用(L) = BaseFee × FeeRate ^ (L - 1)
```
- `BaseFee = 1000` 金币，`FeeRate = 1.2`

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/chest/info` | 获取箱子数量、区域等级 |
| POST | `/api/chest/open` | 开箱 |
| POST | `/api/chest/upgrade_zone` | 升级开箱区域 |

### 3.6 战斗校验模块

**功能**：

| 方案 | 说明 | 时机 |
|------|------|------|
| **方案A** 合理性校验 | 检查通关时间、总伤害是否在玩家战力允许范围内 | 收到上报后立即执行（< 100ms） |
| **方案B** 服务端重跑 | 服务端用相同输入跑简化版战斗，比对关键指标 | 方案A 通过后异步执行（后台队列） |

**方案A 合理性边界**：

| 指标 | 校验规则 |
|------|----------|
| 通关时间 | 不低于关卡理论最短时间 × 0.8 |
| 总伤害 | 不超过玩家 DPS × 关卡时长 × 1.2 |
| 死亡判断 | 玩家 HP 不能在 BOSS 前归零 |
| 波次顺序 | 必须按 1→2→3→4→BOSS 顺序 |

**方案B 简化版关键帧**：
- T0：战斗开始（加载玩家属性 + 技能 + 装备 + 关卡参数）
- T1~T4：每波小怪生成时刻 — 计算该波次净伤害（DPS × 波次时长 − 怪物回血）
- T5：BOSS 出现 — 逐一模拟技能释放，计算技能伤害
- T6：战斗结束 — 比对最终 HP 变化

**差异容差**：服务端模拟的 HP 变化与客户端上报差值 < 5%，视为合法。

**WebSocket 交互流程**：
```
客户端                          服务端
  │                               │
  │── WS 连接 ──────────────────→ │
  │←─ 连接确认 ────────────────── │
  │                               │
  │── 开始战斗（关卡参数请求）───→ │
  │←─ 下发关卡配置 ────────────── │
  │                               │
  │   [客户端本地战斗...]          │
  │                               │
  │── 上报战斗摘要 ──────────────→ │
  │   {伤害,时间,波次,...}         │
  │                          方案A │
  │←─ 方案A 结果（即时）───────── │
  │                          方案B │
  │←─ 方案B 结果（1~3秒后）────── │
  │   {是否合法, 差异率}          │
  │                               │
```

**关键接口**：

| 协议 | 路径 | 说明 |
|------|------|------|
| WebSocket | `/ws/battle` | 战斗上报与校验 |

**上报摘要数据结构**：
```json
{
  "stage_id": "1-3",
  "total_damage_dealt": 12500,
  "total_damage_taken": 3200,
  "clear_time_ms": 85000,
  "waves": [
    {"wave": 1, "kills": 5, "damage": 2000},
    {"wave": 2, "kills": 5, "damage": 2200},
    {"wave": 3, "kills": 5, "damage": 2500},
```

### 3.7 关卡模块

**功能**：
- 关卡配置管理：服务端存储关卡参数（怪物属性、数量、布置），客户端请求后下发
- 进度记录：记录玩家已通关的最高关卡，未通关不可跳关
- 关卡奖励：通关后发放金币和箱子

**关卡配置结构**（服务端定义）：
```json
{
  "stage_id": "1-3",
  "chapter": 1,
  "level": 3,
  "waves": [
    {"wave": 1, "monsters": [
      {"type": "slime", "count": 3, "hp": 200, "atk": 30, "def": 10}
    ]},
    {"wave": 5, "monsters": [
      {"type": "dragon_boss", "count": 1, "hp": 2000, "atk": 80, "def": 30}
    ], "is_boss": true}
  ]
}
```

**按常规做法 — 关卡配置服务端统一下发**，理由：
- 防止客户端篡改关卡参数
- 策划可热更新关卡数据，无需发版
- 服务端校验时使用同一份配置

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/stage/start?stage_id=1-3` | 获取关卡配置 |
| GET | `/api/stage/progress` | 获取进度 |
| POST | `/api/stage/complete` | 通关结算（发奖励） |

### 3.8 排行榜模块

**排名规则**：
- 主排序：已通关最高章节（降序）→ 同章内最高小关（降序）
- 次级排序：该关通关时间（升序，即越快越前）
- 范围：按服务器分区，每个分区独立排行

**刷新策略**：
- **实时刷新**：玩家每次通关校验通过后，立即更新排行榜
- 实现方式：Redis Sorted Set，key 为 `leaderboard:{分区ID}`

**关键接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/leaderboard?page=1&size=50` | 获取排行榜（分页） |
| GET | `/api/leaderboard/my_rank` | 获取我的排名 |

### 3.9 货币模块

**货币类型**：

| 货币 | 存储字段 | 获取途径 | 主要用途 |
|------|----------|----------|----------|
| 金币 | `gold` | 关卡掉落、装备分解 | 升级开箱区域 |
| 技能券 | `skill_tickets` | 关卡奖励、每日登录 | 技能商店抽取 |

**设计原则**：
- 货币变动为原子操作（数据库行锁 / UPDATE SET `gold = gold + N`）
- 记录流水表（`currency_logs`）用于对账和排查

## 四、数据库设计概要

### 4.1 核心表

```sql
-- 账号表
accounts (
    id            BIGSERIAL PRIMARY KEY,
    phone         VARCHAR(20) UNIQUE,
    email         VARCHAR(255) UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    created_at    TIMESTAMP DEFAULT NOW()
)

-- 角色表
characters (
    id          BIGSERIAL PRIMARY KEY,
    account_id  BIGINT REFERENCES accounts(id) UNIQUE,
    nickname    VARCHAR(50) NOT NULL,
    level       INT DEFAULT 1,
    exp         BIGINT DEFAULT 0,
    base_atk    INT DEFAULT 10,
    base_def    INT DEFAULT 5,
    base_hp     INT DEFAULT 100,
    class       VARCHAR(20) DEFAULT 'warrior',
    gold        BIGINT DEFAULT 0,
    skill_tickets INT DEFAULT 0,
    created_at  TIMESTAMP DEFAULT NOW()
)

-- 技能仓库（JSONB 存储）
skill_inventory (
    id          BIGSERIAL PRIMARY KEY,
    character_id BIGINT REFERENCES characters(id),
    skills      JSONB DEFAULT '[]'::jsonb,
    -- skills: [{"skill_id":"s001","quality":"rare","level":1,"count":3,"obtained_at":"..."}]
    shop_level  INT DEFAULT 1,        -- 技能商店等级（最高 28）
    total_pulls INT DEFAULT 0,        -- 累计抽取次数（用于商店升级）
    equipped    JSONB DEFAULT '[]'::jsonb
    -- equipped: ["s001","s003","s005","s007"] (最大 4 个)
)

-- 装备仓库（JSONB 存储）
equipment_inventory (
    id           BIGSERIAL PRIMARY KEY,
    character_id BIGINT REFERENCES characters(id),
    items        JSONB DEFAULT '[]'::jsonb,
    -- items: [{"uid":"e-xxx","slot":"weapon","quality":"epic","atk":45,"def":0,"hp":20}]
    equipped     JSONB DEFAULT '{}'::jsonb
    -- equipped: {"weapon":"e-xxx","helmet":"e-yyy",...}
)

-- 箱子背包
chest_inventory (
    id           BIGSERIAL PRIMARY KEY,
    character_id BIGINT REFERENCES characters(id) UNIQUE,
    chest_count  INT DEFAULT 0,        -- 未开箱数量
    zone_level   INT DEFAULT 1         -- 开箱区域等级
)

-- 关卡进度
stage_progress (
    id           BIGSERIAL PRIMARY KEY,
    character_id BIGINT REFERENCES characters(id) UNIQUE,
    chapter      INT DEFAULT 1,
    level        INT DEFAULT 1,         -- 当前最高已通关的小关
    updated_at   TIMESTAMP DEFAULT NOW()
)

-- 货币流水（审计用）
currency_logs (
    id           BIGSERIAL PRIMARY KEY,
    character_id BIGINT REFERENCES characters(id),
    currency     VARCHAR(20),           -- 'gold' | 'skill_ticket'
    amount       BIGINT,                -- 正数为收入，负数为支出
    reason       VARCHAR(100),          -- 'stage_reward' | 'decompose' | 'gacha' | 'zone_upgrade'
    created_at   TIMESTAMP DEFAULT NOW()
)
```

### 4.2 索引设计

| 表 | 索引 | 用途 |
|----|------|------|
| `accounts` | `phone, email` | 登录查询 |
| `characters` | `account_id` | 角色关联 |
| `stage_progress` | `chapter DESC, level DESC` | 排行榜查询 |
| `currency_logs` | `character_id, created_at` | 流水查询 |

## 五、模块间关系与数据流

### 5.1 核心业务流程

```
┌─────────┐   登录注册    ┌─────────┐
│ 账号模块 │ ←──────────→ │ 角色模块 │
└─────────┘              └────┬────┘
                              │ 角色属性
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
         ┌─────────┐    ┌─────────┐    ┌─────────┐
         │ 装备模块 │    │ 技能模块 │    │ 关卡模块 │
         └────┬─────┘    └────┬─────┘    └────┬─────┘
              │               │               │
              │ 装备加成      │ 技能配置       │ 关卡参数
              └───────────────┼───────────────┘
                              ▼
                      ┌─────────────┐
                      │  战斗模块    │
                      │ (客户端执行) │
                      └──────┬──────┘
                             │ 战斗摘要 + WebSocket
                             ▼
                      ┌─────────────┐
                      │ 战斗校验模块 │
                      │  方案A + B   │
                      └──┬──────┬───┘
                         │      │
                    ┌────▼┐ ┌──▼──────┐
                    │ 货币 │ │排行榜模 │
                    │ 模块 │ │   块    │
                    └─────┘ └─────────┘
```

### 5.2 关键数据流

**战斗结算完整流程**：
1. 客户端请求关卡配置 → 关卡模块返回（含怪物属性/布置）
2. 客户端本地战斗 → 生成战斗摘要
3. 客户端通过 WebSocket 上报摘要 → 战斗校验模块
4. 方案A 即时返回（秒级）→ 方案B 异步后台执行
5. 校验通过 → 调用关卡模块更新进度 → 调用货币模块发奖励（金币+箱子+技能券，不含经验）
6. 关卡模块通知排行榜模块刷新排名
7. 校验结果通过 WebSocket 返回客户端

**装备分解流程**：
1. 客户端请求分解 → 装备模块从仓库移除 → 按品质产出经验和金币
2. 调用角色模块增加经验（可能升级）→ 调用货币模块增加金币

**技能抽取流程**：
1. 客户端请求抽取 → 技能模块扣除技能券 → 按概率随机生成技能
2. 技能写入仓库 → 更新累计抽取次数和商店等级

## 六、技术决策

| 决策项 | 方案 | 理由 |
|--------|------|------|
| Token 方案 | JWT 双 Token（Access + Refresh） | 游戏行业主流方案，无状态可扩展 |
| 关卡配置 | 服务端统一下发 | 防篡改，支持策划热更新 |
| 战斗上报 | 战斗结束后一次性上报摘要 | 减少网络开销，方案A/B 不依赖实时数据 |
| 方案B 关键帧 | 波次节点 + 技能释放节点 | 平衡校验精度与性能 |
| 方案B 容差 | 5% HP 差异 | 覆盖客户端浮点/帧率差异 |
| 排行榜实现 | Redis Sorted Set + 分区 | 实时排序性能好，分区隔离 |
| 装备/技能存储 | PostgreSQL JSONB | 灵活存储变长列表，支持索引查询 |
| 保底机制 | 无保底 | 简化设计，依靠商店等级概率梯度激励 |
| 开箱品质池 | 最多 4 品质旋转门，满级 28 固定 5% | 保证高等级后仍可获取中品质装备，满级给予稳定预期 |
